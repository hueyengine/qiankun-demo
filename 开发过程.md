# qiankun-demo 开发过程梳理

本文档记录从「零」到「跑通」整个 qiankun 微前端 Demo 的设计与实施过程，包含每一步做了什么、为什么这么做、踩了哪些坑以及最终如何落地。`README.md` 是给使用者看的「使用文档」，本文档是给作者/复盘者看的「开发笔记」。

> 项目最终形态：一个 Vue2 + Webpack 主应用（基座），承载三个子应用（1 个 Vue2 + Webpack、2 个 Vue3 + Vite）。预留 React + Vite 作为扩展位。

---

## 一、需求与目标

最初目标是「学习并实践 qiankun」，做一个能跑、能体现典型问题的 Demo，并写到 README 里。在跟用户来回确认后，把目标具体化为以下几条：

1. 主应用作为基座，只渲染顶部导航；左侧菜单和内容区都由子应用自己实现。
2. 主子应用技术栈刻意做混搭，模拟真实「巨石→微前端」的中间状态：
   - 主应用 = Vue2 + Webpack
   - 子应用一 = Vue2 + Webpack
   - 子应用二、三 = Vue3 + Vite（两个，用来观察「同构」时的共享依赖与隔离问题）
   - React + Vite 作为扩展位（README 里讨论，本仓库暂不实现）
3. 主子应用都接入 Element UI / Element Plus，验证组件库（含 Dialog 等弹层）在微前端下表现正常。
4. 必须开启 `experimentalStyleIsolation`，并且要真的隔离住，不允许子应用样式互相污染。
5. 根目录提供一键启动脚本与 `.gitignore`，README 里覆盖共享依赖、巨石迁移等进阶话题。

## 二、方案确认（先方案、后代码）

为了避免「一边写一边改」，先把方案讨论清楚。这一阶段没有写代码，只产出 README。

### 2.1 技术栈迭代

技术栈经过几轮调整：

1. 第一轮提议：「主应用 Vue3 + Vite，子应用一个 Vue2、一个 React」。
2. 用户提出技术栈调整为「主应用 Vue2 + Webpack、3 个子应用（1 个 Vue2 + Webpack、2 个 Vue3 + Vite），React + Vite 作为扩展」。这才是最终采纳的方案。

这套搭配的好处：

- 主应用 Webpack 是企业里巨石老应用的常见形态，符合「老系统改基座」的真实场景。
- 同时存在 Vue2 和 Vue3，能直接体现框架运行时无法共享的现实约束。
- 两个 Vue3 + Vite 子应用，则用来讨论「同构子应用之间能否共享依赖」。

### 2.2 应用布局

布局也调整过一次：

- 第一版：主应用提供顶部导航 + 左侧菜单。
- 最终版：**主应用只提供顶部导航**；不同顶部导航项对应不同子应用；**左侧菜单 + 内容区都属于子应用自己**。

这种切法更贴近实际：左侧菜单往往跟具体业务强绑定，放在子应用里随业务一起演进，比放在基座更合理。

### 2.3 进阶话题

在确定方案的过程中，跟用户讨论并写进 README 的进阶话题包括：

- 共享依赖：框架运行时（Vue2/Vue3/React）跨技术栈无法共享，多个 Vue3 子应用之间是否应共享 Vue/Element Plus 等。
- 巨石迁移：旧应用接入 qiankun 前需要做的清理（全局副作用、定时器、事件监听、样式裸写等）。
- 沙箱（experimentalStyleIsolation / strictStyleIsolation / JS 沙箱）。
- 基座与子应用的契约（生命周期、`activeRule`、容器、`publicPath`、entry）。
- URL/网关/鉴权（history mode 下的 fallback、跨域、统一登录）。

这些话题最终都进了 `README.md` 的「进阶」章节。

## 四、代码生成步骤

下面按实际操作顺序记录。

### 4.1 用脚手架创建四个工程

- `main/`、`sub-vue2/`：用 Vue CLI 创建 Vue2 工程。这一步最初尝试 `pnpm` 失败，最后用 `--packageManager npm` 显式指定 npm 才稳定下来。
- `sub-vue3-a/`、`sub-vue3-b/`：用 `npm create vite` 创建 Vue3 工程。

实际命令：

```bash
# 在仓库根目录执行
# 1) 主应用 / Vue2 子应用：Vue CLI（Vue2 预设）
npx @vue/cli create main --packageManager npm --preset __default_vue_2__
npx @vue/cli create sub-vue2 --packageManager npm --preset __default_vue_2__

# 2) Vue3 + Vite 子应用：Vite 官方脚手架
npm create vite@latest sub-vue3-a -- --template vue
npm create vite@latest sub-vue3-b -- --template vue
```

> 备注：第一次创建 `main` 时用了默认的 pnpm，被卡住了；后来加 `--packageManager npm` 才稳定。

四个工程都是各自独立的子项目，**没有用 monorepo 工具**（pnpm workspace / yarn workspaces 都没用），就是普通的多目录并列。原因是 qiankun 本身并不要求 monorepo，且想刻意还原「不同团队不同仓库」的现实情况。

### 4.3 主应用接入 qiankun

主应用只做三件事：渲染顶部 Shell、提供 `#subapp-viewport` 容器、注册 + 启动 qiankun。

- `main/vue.config.js`：开 `historyApiFallback`、固定 8080。
- `main/src/router/index.js`：所有 `/app/**` 都路由到 `ShellLayout`，让基座 Vue Router 不去抢子应用的子路径。
- `main/src/layouts/ShellLayout.vue`：用 `el-menu` 实现顶部导航，下面只放一个 `<div id="subapp-viewport">`。
- `main/src/micro/index.js`：注册三个子应用并 `start({ sandbox: { experimentalStyleIsolation: true } })`。
- `main/src/main.js`：注册 ElementUI、调用 `setupQiankun()`、挂载根实例。

关键代码：

```1:30:main/src/micro/index.js
import { registerMicroApps, start } from 'qiankun'

export function setupQiankun() {
  registerMicroApps([
    {
      name: 'sub-vue2',
      entry: '//localhost:8081',
      container: '#subapp-viewport',
      activeRule: (location) => location.pathname.startsWith('/app/vue2'),
    },
    {
      name: 'sub-vue3-a',
      entry: '//localhost:8082',
      container: '#subapp-viewport',
      activeRule: (location) => location.pathname.startsWith('/app/vue3-a'),
    },
    {
      name: 'sub-vue3-b',
      entry: '//localhost:8083',
      container: '#subapp-viewport',
      activeRule: (location) => location.pathname.startsWith('/app/vue3-b'),
    },
  ])

  start({
    sandbox: {
      experimentalStyleIsolation: true,
    },
  })
}
```

### 4.4 Vue2 子应用接入

Webpack 子应用接 qiankun 是「教科书写法」：

1. `sub-vue2/src/public-path.js`：被 qiankun 加载时改写 `__webpack_public_path__`，否则静态资源会从基座域名加载、404。
2. `sub-vue2/vue.config.js`：把输出改为 UMD（`libraryTarget: 'umd'`、自定义 `library` 名、`globalObject: 'window'`），并允许跨域（`Access-Control-Allow-Origin: *`）。
3. `sub-vue2/src/router/index.js`：`base` 根据 `window.__POWERED_BY_QIANKUN__` 切换为 `/app/vue2/` 或 `/`，保证「独立运行 / 微前端」两种形态都能跑。
4. `sub-vue2/src/main.js`：实现 `bootstrap` / `mount` / `unmount` 三个生命周期；mount 时按 `props.container` 的真实 DOM 找 `#app` 挂载，unmount 时 `$destroy()` + 摘节点，防止内存泄漏。

```31:39:sub-vue2/src/main.js
export async function unmount() {
  if (instance) {
    instance.$destroy()
    if (instance.$el && instance.$el.parentNode) {
      instance.$el.parentNode.removeChild(instance.$el)
    }
    instance = null
  }
}
```

### 4.5 Vue3 + Vite 子应用接入

Vite 子应用没有 UMD 包模式，直接用 Webpack 的写法是接不进来的。这里用 `vite-plugin-qiankun` 解决：

```9:13:sub-vue3-a/vite.config.js
export default defineConfig({
  plugins: [
    vue(),
    qiankun(APP_NAME, { useDevMode: true }),
  ],
```

`main.js` 用 `renderWithQiankun` 注册生命周期；`router/index.js` 同样根据 `qiankunWindow.__POWERED_BY_QIANKUN__` 切 `base`。

```22:41:sub-vue3-a/src/main.js
renderWithQiankun({
  bootstrap() {
    return Promise.resolve()
  },
  mount(props) {
    render(props)
  },
  unmount(props) {
    releaseStyles()
    if (app) {
      app.unmount()
      app = null
    }
    const { container } = props || {}
    if (container) {
      const el = container.querySelector('#app')
      if (el) el.innerHTML = ''
    }
  },
})
```

`sub-vue3-b` 与 `sub-vue3-a` 完全对称，仅端口、应用名、`router base`、scope 类不同。

### 4.6 UI 组件库

应用户要求引入：

- 主应用与 sub-vue2：`element-ui`（Vue2 版）。
- sub-vue3-a / sub-vue3-b：`element-plus`（Vue3 版）。

- Vue2 用 `:append-to-body="false"`。
- Vue3 用 `:teleported="false"`。

### 5.3 样式隔离（核心难点）

这是花时间最多的一步。先看现象：

- 主应用 `start` 已经开了 `experimentalStyleIsolation: true`。
- 在 sub-vue3-a 写了 `.demo-title { color: green }`，sub-vue3-b 写了 `.demo-title { color: purple }`，期望切换互不影响。
- 实际：第一次进 vue3-a 是绿色没问题；切到 vue3-b 是紫色也没问题；**再切回 vue3-a，颜色变成了紫色**——后挂载的子应用样式吃掉了前面那个。

#### 为什么 `experimentalStyleIsolation` 没生效

`experimentalStyleIsolation` 的实现思路是：qiankun 给子应用容器打一个 `data-qiankun="子应用名"` 属性，并把它接管到的 `<style>` / `<link>` 选择器在「插入容器时」自动加上 `div[data-qiankun~="子应用名"]` 前缀。

问题在于 **Vite dev 模式下，Vue/Vite 的样式是通过 JS 动态注入到 `document.head` 的**，不是放在子应用 entry 的 HTML 里——qiankun 接管不到这些 style 节点，自然就没法给它们加前缀，于是这些样式就以「全局」身份留在 `document.head`，谁后插入谁覆盖谁。

#### 解决思路（保留 `experimentalStyleIsolation`）

用户明确要求「不要换策略，就是要 `experimentalStyleIsolation`」。所以方案是「让 `experimentalStyleIsolation` 的前缀机制名义上还在，但样式自己先把作用域写好，另外把样式标签也搬运到子应用容器里」。

先装兜底用的 PostCSS 插件：

```bash
npm i -D postcss-prefix-selector --prefix sub-vue3-a
npm i -D postcss-prefix-selector --prefix sub-vue3-b
```

两步同时做：

1. **编译期前缀（postcss-prefix-selector）**

   在 `vite.config.js` 里把所有 CSS 选择器前面统一拼上 `div[data-qiankun~="sub-vue3-a"]`（sub-vue3-b 同理），这样 CSS 即便被注入到 `document.head` 也只会作用于带这个属性的容器——这跟 `experimentalStyleIsolation` 给容器添加的属性完全对得上。

   ```9:28:sub-vue3-a/vite.config.js
   export default defineConfig({
     plugins: [
       vue(),
       qiankun(APP_NAME, { useDevMode: true }),
     ],
     css: {
       postcss: {
         plugins: [
           prefixer({
             prefix: SCOPE,
             transform(prefix, selector, prefixedSelector) {
               if (/^(html|:root|body)\b/.test(selector)) {
                 return selector.replace(/^(html|:root|body)/, prefix)
               }
               return prefixedSelector
             },
           }),
         ],
       },
     },
   ```

   `transform` 里特地处理了 `html / :root / body`：直接整体替换为 scope，不再保留全局选择器，避免子应用偷偷改全局。

2. **运行期搬运（style-hijack.js）**

   光靠编译期前缀还不够。Vite 在卸载时不会主动回收已经注入到 `document.head` 的 style 标签——卸载子应用后这些标签仍留在主文档里，下次再 mount 又会插一份新的，越积越多。所以再加一个轻量「搬运工」：在 `mount` 时把本子应用注入的 style 节点从 `document.head` 搬到 `props.container` 内部，并用 `MutationObserver` 持续把后续新增的也接力搬运过去；`unmount` 时关闭 observer。

   ```1:46:sub-vue3-a/src/style-hijack.js
   const preHijackNodes = new WeakSet()
   Array.from(document.head.childNodes).forEach((n) => preHijackNodes.add(n))

   const ownedNodes = []
   let observer = null
   let initialized = false

   function isOwnedStyleNode(node) {
     if (!node || node.nodeType !== 1) return false
     if (preHijackNodes.has(node)) return false
     if (node.tagName === 'STYLE') return node.hasAttribute('data-vite-dev-id')
     if (node.tagName === 'LINK' && node.rel === 'stylesheet') return true
     return false
   }

   export function hijackStyles(container) {
     if (!container) return

     if (!initialized) {
       Array.from(
         document.head.querySelectorAll('style, link[rel="stylesheet"]')
       ).forEach((node) => {
         if (!isOwnedStyleNode(node)) return
         ownedNodes.push(node)
         container.appendChild(node)
       })
       initialized = true
     } else {
       ownedNodes.forEach((node) => {
         if (node.parentNode !== container) container.appendChild(node)
       })
     }

     observer?.disconnect()
     observer = new MutationObserver((mutations) => {
       for (const m of mutations) {
         m.addedNodes.forEach((node) => {
           if (!isOwnedStyleNode(node)) return
           if (node.parentNode === container) return
           if (!ownedNodes.includes(node)) ownedNodes.push(node)
           container.appendChild(node)
         })
       }
     })
     observer.observe(document.head, { childList: true })
   }
   ```

   `isOwnedStyleNode` 的两个细节：
   - 启动时记录 `document.head` 的现有节点到 `preHijackNodes`，搬运时跳过——避免把基座自己的样式偷走。
   - 只认两类：带 `data-vite-dev-id` 的 `<style>`（Vite dev 注入）和 `link[rel="stylesheet"]`（生产/CSS link）。

3. **`main.js` 串起来**：mount 时调 `hijackStyles(container)`，unmount 时 `releaseStyles()`。

#### 这一步的最终效果

- `experimentalStyleIsolation` 这个配置保留了，没被换掉。
- 子应用的 CSS 在编译期就带上了独有前缀，多个 Vue3 子应用之间互不污染。
- 样式标签也跟随子应用进出容器，符合「卸载即清场」的直觉。

验证命令：

```bash
# 改完 vite.config.js 后，先重启 dev 让 postcss 配置生效
npm run dev --prefix sub-vue3-a
npm run dev --prefix sub-vue3-b

# 想直接看打包后的 CSS 是不是真的带了前缀，可以构建一次然后翻 dist 里的 .css
npm run build --prefix sub-vue3-a
# 期望在 dist 里看到 div[data-qiankun~="sub-vue3-a"] .demo-title { ... } 这样的选择器
```

补充收尾：把 `Home.vue` 里用来做颜色验证的样式块也都改成了 `<style scoped>`，进一步降低组件级冲突的概率。

## 七、本地启动

依赖只需在每个工程目录单独装一次：

```bash
npm i --prefix main
npm i --prefix sub-vue2
npm i --prefix sub-vue3-a
npm i --prefix sub-vue3-b
npm i
```

启动方式（任选其一）：

```bash
# 一键全起
npm run dev:all

# 或分别起，调试时更直观
npm run dev:main
npm run dev:sub-vue2
npm run dev:sub-vue3-a
npm run dev:sub-vue3-b
```

打开 `http://localhost:8080`，顶部菜单切换三个子应用，观察：

1. 子应用左侧菜单和内容区都来自子应用本身。
2. 反复切换 vue3-a ↔ vue3-b，`.demo-title` 颜色不会互相污染。
3. Element UI/Plus 的 Dialog 弹在子应用容器里，不会从主应用根挂出。

至此整个 Demo 落地完成。

## 附录 A · 命令速查

下面把整套流程涉及的命令按阶段汇总，从空目录开始照抄即可。

### A.1 创建工程

```bash
# 进入仓库根目录后：
npx @vue/cli create main --packageManager npm --preset __default_vue_2__
npx @vue/cli create sub-vue2 --packageManager npm --preset __default_vue_2__
npm create vite@latest sub-vue3-a -- --template vue
npm create vite@latest sub-vue3-b -- --template vue
```

### A.2 安装运行时依赖

```bash
# 主应用
npm i qiankun vue-router@3 element-ui --prefix main

# Vue2 子应用
npm i vue-router@3 element-ui --prefix sub-vue2

# Vue3 + Vite 子应用 A
npm i vue-router@4 element-plus vite-plugin-qiankun --prefix sub-vue3-a
npm i -D postcss-prefix-selector --prefix sub-vue3-a

# Vue3 + Vite 子应用 B
npm i vue-router@4 element-plus vite-plugin-qiankun --prefix sub-vue3-b
npm i -D postcss-prefix-selector --prefix sub-vue3-b

# 根目录：一键启动
npm i -D concurrently
```

### A.3 日常开发

```bash
# 一次起所有应用
npm run dev:all

# 或者按需起
npm run dev:main
npm run dev:sub-vue2
npm run dev:sub-vue3-a
npm run dev:sub-vue3-b
```
